# i18n & Accessibility

The app ships **English and Albanian** (`en`/`sq`, business doc §9.1) via **react-i18next**, and every screen is keyboard- and screen-reader-accessible. User-facing copy is translated, not hardcoded.

## i18n
```tsx
const { t } = useTranslation("predictions");
return <button>{t("submit")}</button>; // not <button>Submit</button>
```

- **No hardcoded user-facing strings.** Wrap copy in `t(...)`; keys live in the feature's namespace.
- Keep keys **semantic**, not string-concatenated; interpolate values (`t("greeting", { name })`), don't build sentences by joining.
- Empty/loading/error copy is translated too ("No upcoming matches" → `t("calendar.empty")`).
- If i18n is explicitly deferred for the MVP (architecture §11), still **centralise copy** in one place rather than scattering literals, so it's a lift-and-shift later.

## Accessibility
- Use **semantic HTML**: `<button>` for actions, `<a>` for navigation, `<label htmlFor>` for inputs. Don't make a `<div>` clickable when a `<button>` fits.
- Form inputs are labelled and errors are associated with `aria-invalid` / `aria-describedby` (see `forms-validation.md`).
- Images have `alt`; decorative images use `alt=""`.
- Interactive custom components expose the right role and full keyboard handling (focus, Enter/Space, Esc for dismissables).
- Loading states are announced where it matters (`aria-busy`, live regions for async results) — never a silent blank screen (see `components-presentation-only.md`).
- Respect colour-contrast and don't convey state by colour alone.
